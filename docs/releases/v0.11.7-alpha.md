# Release v0.11.7-alpha

**Date** : 13 mai 2026
**Mode** : Complet (audit 3 agents + arbitrage Claude Code)
**Version precedente** : v0.11.6-alpha (30 avril 2026)

## Contexte

Bugs remontes par Smileshoot (testeur alpha actif) sur Discord #bugs le 10 mai a 09h26 et 09h29 UTC. Detection via l'agent Katia (OpenClaw VPS Agents) qui scrute les canaux #bugs, #suggestions et #discussion en continu.

## Fixes inclus (PR #75)

| Ref | Type | Description | Fichiers |
|-----|------|-------------|----------|
| BUG-096 | fix(frontend) | 66 sequences `\u00xx` litterales dans des JSX text nodes affichaient le code brut a l'ecran ("Payée" au lieu de "Payee" / "Payée"). Cause : generation initiale des fichiers avec JSON encoding au lieu d'UTF-8 direct. Solution : conversion totale via script Python sur les 4 fichiers concernes. Test de regression scanne tout le frontend `.ts`/`.tsx`. | `InvoiceForm.tsx`, `InvoicesPanel.tsx`, `PromptLibrary.tsx`, `services/api/prompts.ts` |
| BUG-097 | fix(frontend) | ActionPanel annoncait "Resultat insere dans le chat" alors qu'aucun code ne realisait l'insertion. UI mensongere. Cause : `actionsStore.refreshTask()` n'importait pas `chatStore` et ne declenchait jamais `addMessage()` lors de la transition vers `completed`. Solution : import `useChatStore`, helper `insertResultInChat()` avec Set d'idempotence, gestion des statuts `completed` et `error`. | `stores/actionsStore.ts` |

**Tests ajoutes** (43 lignes) :
- `test_no_unicode_escape_in_jsx_text_nodes` : scan recursif du frontend, fail si une sequence `\u00XX` est trouvee hors fichiers `.test.ts`
- `test_actions_store_inserts_result_in_chat_on_completion` : verifie que `actionsStore.ts` importe `chatStore`, contient le marqueur `BUG-097` et appelle `addMessage()`

## Audit multi-agents (mode complet)

### Agent 1 - Securite / Web

**Verdict : PASS** (2 warnings mineurs)

- Pas de risque XSS : chaines constantes UI, React echappe par defaut
- Pas de double-encoding : caracteres en NFC, identiques byte-pour-byte aux sequences d'origine
- Cross-store `useChatStore.getState()` est un pattern tolere Zustand (synchrone, pas de stale state)
- `task.result` rendu via `ReactMarkdown` avec filtrage des schemes non-http(s) par defaut
- 0 CVE connue sur les libs touchees

### Agent 2 - UX/UI

**Verdict : PASS** (1 warning cosmetique)

- BUG-096 : tous les labels critiques (Payee, Envoyee, Emis, Validite, Echeance...) lisibles correctement
- BUG-097 : feedback enfin coherent, l'utilisateur voit le resultat dans le chat
- Warning : pas de distinction visuelle "message d'action" vs "message Claude classique" → backlog BUG-098

### Agent 3 - Avocat du diable

**Verdict initial : WARNING** (2 critiques, 3 moyens, 2 mineurs)

Apres verification par Claude Code :

| Objection | Statut | Action |
|-----------|--------|--------|
| Re-injection au reload (re-fetch de tache completed) | Faible : `actionsStore` n'a pas de middleware `persist` | Renforce via Set d'idempotence |
| Conv active differente entre lancement et completion | Edge case reel mais probabilite faible (1 testeur actif, 1 conv typique) | Backlog BUG-098 |
| Status `error` silencieux | Valable | **Corrige avant merge** : insertion d'un message d'erreur formate |
| Regex test trop etroite (`\u00XX` seul) | Mineur | Backlog BUG-098 |
| Race 2 polling concurrents | Mineur | Resolu par Set d'idempotence |

## Decision Claude Code (arbitre)

**MERGER avec 2 renforcements ajoutes avant merge** :

1. **Set `insertedTaskIds`** : defense en profondeur contre les doublons (re-poll concurrent, futur re-fetch historique)
2. **Gestion `error`** : insertion d'un message "Erreur : <error>" formate avec resultat partiel s'il existe

Le fix initial corrigeait deja les 2 bugs Smileshoot. Les renforcements ferment 2 angles morts soulevés par l'avocat du diable sans bloquer la release.
