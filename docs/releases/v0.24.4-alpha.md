## Bugs corrigés

### BUG-108 — boucle lecture des mails (lcjp, Mistral, Windows)
- **Cause racine** : `providers/base.py:_append_openai_tool_turn` posait `content = assistant_content or None` ; quand le modèle écrivait du texte avant l'appel d'outil, Mistral rejetait le message (400 : content non vide + tool_calls interdits), le résultat de `read_emails` était perdu → boucle « Max tool iterations » → fausse erreur d'auth inventée.
- **Fix** : un message porteur de `tool_calls` envoie toujours `content=None` (valeur canonique OpenAI/litellm, déjà éprouvée côté Mistral). Couvre aussi OpenRouter/Infomaniak routant vers Mistral.
- **Test** : `tests/test_provider_tools.py::test_mistral_tool_turn_content_vide_pas_de_400_bug108` (fake client 400 sur content non vide + tool_calls).

### BUG-109 + boucle « connexion expirée » (lcjp, calendrier Google)
- **Cause racine** : le fix v0.24.1 (fcc9b48) avait mis le mot « reconnecte » dans le message 403, capté par l'heuristique du `CalendarPanel` (`includes('reconnect')`) → un vrai 403 Google (API Calendar non activée) était classé comme jeton expiré → bannière jaune trompeuse, message actionnable masqué (`error && !needsReauth`), et boucle de reconnexion infinie (l'OAuth réussit mais ne réactive pas l'API).
- **Fix** : backend, message 403 reformulé sans « reconnecte » (reste actionnable) ; frontend, helper testable `classifyCalendarError` qui affiche le message actionnable sur un 403 ET **désarme** `needsReauth` (un 403 prouve que le jeton est valide ; `needsReauth` est partagé avec l'email et une expiration Gmail antérieure pouvait entretenir la boucle — défaut identifié en revue adversariale et corrigé). « reconnect » reste détecté pour la vraie expiration backend (« Please reconnect your account »).
- **Tests** : `tests/test_routers_calendar.py` (assertion « reconnect » absent du 403) + `calendarErrors.test.ts` (7 cas).

### BUG-107 — régression historique chat (Capov, macOS)
- **Cause racine** : une conversation créée par un agent (préparation de RDV) est locale, non synchronisée. Le clic sidebar interrogeait le backend avec un ID inconnu → `[]` (HTTP 200) → `setConversationMessages` écrasait le message et marquait `synced:true` → disparition de l'historique.
- **Fix** : `chatStore.setConversationMessages` ne vide plus une conversation non synchronisée avec une liste vide ; `ConversationSidebar` ne fetch que si la conv est synced et vide (même garde que `useConversationSync`, BUG-068).
- **Test** : `chatStore.test.ts` (2 cas de régression).
