# Release THÉRÈSE v0.24.3-alpha

**Date** : 11/06/2026 (soir)
**Type** : PATCH (1 correctif transversal de layout)
**Mode** : fast-track (commit direct sur `main`, pas de PR)

## Contexte

Ludo a constaté que le bouton « Générer une réponse » de l'univers Email
restait invisible **en 0.24.2** : le fix `shrink-0` de la 0.24.2 traitait un
symptôme interne au panneau, pas la vraie cause.

## Cause racine réelle (prouvée par reproduction)

Les 6 panneaux standalone (email, CRM, calendrier, tâches, factures, mémoire)
avaient une racine `h-full` = 100 % du conteneur de vue de `ChatLayout` — qui
contient AUSSI la back-bar « ← Chat » (~37 px). Le panneau débordait donc de
cette hauteur sous la fenêtre (`overflow-hidden` rognait le bas). Visible
surtout dans l'univers Email (barre d'actions ancrée en bas) ; latent ailleurs
(~30 derniers px de contenu inaccessibles derrière une zone scrollable).

**Preuve** : repro HTML de la chaîne de conteneurs mesurée dans Chrome
headless — variante `h-full` : bouton 13 px hors écran ; variante
`flex-1 min-h-0` : bouton visible. (`/tmp/repro-email-layout.html`, captures
du 11/06.)

## Correctif

`fix(ui)` `1896d2b` : `h-full` → `flex-1 min-h-0` sur les 6 racines standalone
(EmailPanel, CRMPanel, CalendarPanel, TasksPanel, InvoicesPanel, MemoryPanel).
2 tests de régression : nouveau test EmailPanel (la racine standalone est
`flex-1 min-h-0`, pas `h-full`) + test MemoryPanel L6 aligné (il verrouillait
littéralement `h-full`).

## Vérifications

- Repro Chrome headless : fix validé visuellement (avant/après).
- Frontend : `tsc` 0 erreur, eslint 0 erreur, **222 tests Vitest verts**.
- Build CI : macOS ARM64, Windows x64, Linux x64 + preflight signature → success.
- Landing + `latest.json` 0.24.3 vérifiés en ligne.

## Leçon (durable)

Un fix de layout ne part plus en release sans **vérification visuelle**
(repro mesurée ou capture) : jsdom/Vitest ne calculent pas le layout, et le
shrink-0 de la 0.24.2 est passé « vert » sans corriger le symptôme réel.
