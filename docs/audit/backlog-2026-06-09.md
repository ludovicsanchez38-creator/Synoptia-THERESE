### Epic : Sécurité & Conformité

#### US-001 : Sortir l'exécution de code du processus backend (sandbox réelle)
**Priorité** : P0 | **Effort** : L | **Axes** : Sécurité | **Source** : SEC1
> En tant qu'utilisateur, je veux que le code généré par le LLM s'exécute dans un processus isolé sans accès à la mémoire du backend, afin qu'une prompt injection (email ou page web piégés) ne puisse pas voler mon session_token ni ma clé Fernet.

**Critères d'acceptation** :
- [ ] `exec()` n'est plus appelé in-process : exécution dans un subprocess dédié (interpréteur séparé, env minimal, sans variables sensibles), avec timeout et limites ressources
- [ ] Les évasions par dunder sont bloquées : `().__class__.__mro__`, `__subclasses__`, `__globals__`, `__builtins__` testées et refusées
- [ ] Test de régression : un payload d'évasion connu (extraction de `session_token` via dunder) retourne une erreur sans rien exécuter ni exposer
- [ ] La regex blocklist actuelle est conservée comme défense en profondeur, pas comme unique barrière

**Notes** : `src/backend/app/services/skills/code_executor.py:43,703-704`. Subprocess + sérialisation JSON ; étudier `multiprocessing.get_context("spawn")` ou sandbox-exec selon OS. Vecteur d'attaque n°1.
